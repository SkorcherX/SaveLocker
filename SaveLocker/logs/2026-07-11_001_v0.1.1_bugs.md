## Bug 1 — Agent UI shows "AGENT V0.0.0"

**Root cause:** MinVer sets `Version`, `FileVersion`, and `AssemblyVersion` via separate MSBuild targets. Passing `--property:Version=$AppVersion` to `dotnet publish` has the highest MSBuild precedence and correctly stamps `Version` and `FileVersion`. However, MinVer still runs its own target and overrides `AssemblyVersion` independently — stamping `0.0.0.0` when git is inaccessible on the GitHub Actions Windows runner.

`UpdateChecker.CurrentVersion` reads `Assembly.GetEntryAssembly()!.GetName().Version`, which returns the `AssemblyVersion` attribute — not `FileVersion`. The `?? new Version(0,1,0)` fallback only fires when the value is null; `0.0.0.0` is not null, so the fallback never activates.

**File:** `installer/build-installer.ps1` — the `dotnet publish` call at line ~40.

**Fix:** Add `"--property:AssemblyVersion=$AppVersion"` alongside the existing `"--property:Version=$AppVersion"`:

```powershell
if ($AppVersion) {
    & $dotnet publish $agentProj -p:PublishProfile=win-x64 `
        "--property:Version=$AppVersion" `
        "--property:AssemblyVersion=$AppVersion" --nologo
} else {
    & $dotnet publish $agentProj -p:PublishProfile=win-x64 --nologo
}
```

**Verify:** After fix, retag and watch the GitHub Actions log for `Agent version:` output. Install the new exe, check the agent tray UI header — should show the correct version, not `0.0.0`.

**Alternative (if above still fails):** Change `UpdateChecker.cs` to read `FileVersionInfo` instead of `AssemblyVersion`. For single-file exes, `Assembly.Location` returns empty string — use `Environment.ProcessPath`:

```csharp
public static readonly Version CurrentVersion =
    Version.TryParse(
        System.Diagnostics.FileVersionInfo
            .GetVersionInfo(Environment.ProcessPath ?? "")
            .FileVersion,
        out var v) ? v : new Version(0, 1, 0);
```

## Bug 2 — No auto-relaunch after silent update

**Root cause:** The `UpdateChecker` invokes the installer with `/SILENT` to suppress the UI. The `[Run]` section in `SaveLocker.iss` previously had the `skipifsilent` flag on the agent launch entry, which suppresses post-install execution when `/SILENT` is passed. So the installer completed silently and the agent was never started.

**File:** `installer/SaveLocker.iss` — `[Run]` section.

**Fix already committed:** `skipifsilent` removed. Current flags: `Flags: nowait postinstall runasoriginaluser`.

**Verify:** This bug cannot be verified without a real end-to-end test:
1. Have an old version of the agent installed and running.
2. Upload a newer installer to the server via the dashboard Configuration → Agent Updates.
3. Wait for the auto-update check (or trigger it manually).
4. Confirm the agent restarts and the tray icon reappears after the update.

## Bug 3 — Uploaded installer lost on every Docker update

**Root cause:** `AgentInstallerService` (`src/Server/Services/AgentInstallerService.cs:20`) resolves the storage root as:

```csharp
_root = cfg["Storage:AgentInstallerRoot"]
    ?? Path.Combine(AppContext.BaseDirectory, "data", "agent-installer");
```

`AppContext.BaseDirectory` inside the container is `/app/` (the publish output directory), so the default path is `/app/data/agent-installer/`. This path is inside the container image filesystem, not on the persistent `/data` volume mount (`/mnt/user/appdata/savelocker → /data`). Every `docker compose up` with a new image wipes it.

`appsettings.json` has `DbPath`, `ArchiveRoot` etc. all pointing to `/data/...` but is missing `AgentInstallerRoot` entirely — leaving the fallback to the in-container path.

**File:** `src/Server/appsettings.json` — `Storage` block.

**Fix:** Add the missing key:

```json
"Storage": {
  "DbPath": "/data/savelocker.db",
  "ArchiveRoot": "/data/archives",
  "AgentInstallerRoot": "/data/agent-installer"
}
```

No code changes needed. The `AgentInstallerService` already reads this key; it just wasn't set in the default config.

**Verify:** After deploying the updated Docker image, upload an installer via the dashboard. Run `docker compose pull && docker compose up -d`. Confirm the installer is still present (the Agent Updates card in Config view should still show the uploaded version).
