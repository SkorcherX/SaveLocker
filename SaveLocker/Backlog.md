- ~~**Linux / Steam Deck hardening: serialize daemon + launch-wrapper state.**~~ ✅ **DONE 2026-07-18.** `Decisions.md` §8. Cross-process per-game sync lock (`AgentStateLock`, flock/share-deny) held *alongside* the existing semaphore, since a flock is per-process and would not separate two threads; PID+GUID temp archive names with a stale sweep; atomic `AtomicFile` writes everywhere; and read-merge-write for config sync state, the offline queue and health events.
  - **The bug this actually fixed:** a daemon writing its stale in-memory `config.json` erased the `LastKnownVersionId` another process had just recorded, so the next push presented a stale parent and the server rejected it — **one machine conflicting with itself**, with nothing in the log to explain it. Reproduced, then fixed.
  - Two incidental fixes fell out: state files (`offline-queue.json`, `health-events.json`) resolved to the *machine default* directory rather than beside the `--config` they belong to, so processes given different configs silently kept private copies; and a one-shot CLI `push` never queued a failed upload at all, so an offline push was simply lost.
  - Covered by `tests/run-concurrency-tests.ps1` (12 checks, cross-OS, in CI) — **verified to fail against the pre-fix code** (4 checks flip), which an earlier racing-pushes version of the test did not. The Linux `run-linux-tests.sh` section pairing the daemon with the **real launch wrapper** is written *and executed*: **33/33 in WSL Ubuntu 24.04**.
  - **Verified on Linux, not only Windows** (2026-07-18, WSL ext4): local-api 15/15 (incl. the `0600` token check Windows skips), concurrency 12/12 (real `flock`, not Windows share-deny), hardening 14/14, health 17/17, enrollment 16/16, Linux harness 33/33.
  - ⏳ **Deferred, as designed:** one state owner via wrapper→daemon IPC over a Unix socket. The locking above makes the current two-owner model correct; IPC would make it simpler. Worth doing before the state files grow further.
- ~~**Linux / Steam Deck hardening: close restore write-through-link and archive-exhaustion gaps.**~~ ✅ **DONE 2026-07-18.** `Decisions.md` §9.
  - **The write-through-link hole was real and proven exploitable.** Against pre-fix code the test overwrites a file *outside* the save folder with attacker-chosen bytes: the target held `linkdir -> /elsewhere`, the archive carried `linkdir/secret.txt`, and `File.Copy` wrote straight through it. Phase 6 made the *delete* pass no-follow but left the *copy* pass following links.
  - Destination paths now refuse to traverse a symlink/junction **below** the save root, and the whole restore is rejected rather than partially applied. **Deviation from the wording above, deliberately:** the target *root* is **followed**, not rejected — a user symlinking their save folder onto an SD card is legitimate and is user-chosen input, unlike the relative paths inside the archive.
  - Zip-bomb limits: 100,000 entries and 2 GB uncompressed, checked against the declared central-directory sizes *and* against bytes actually written (the declared size is attacker-controlled and may lie). Overridable via `SAVELOCKER_MAX_RESTORE_ENTRIES` / `SAVELOCKER_MAX_RESTORE_MB`, which is also what makes the caps testable without a 2 GB fixture.
  - Extraction is now hand-rolled (`ExtractChecked`) to enforce the byte cap, so it re-implements the zip-slip rejection `ZipFile.ExtractToDirectory` gave for free. The existing zip-slip check was kept and re-aimed at the replacement.
  - `run-hardening-tests.ps1` is now **27 checks on Windows / 28 on Linux**, and **verified to fail 7 checks against pre-fix code** on both. Full Linux sweep green: hardening 28/28, concurrency 12/12, local-api 15/15, health 17/17, Linux harness 33/33.

## Medium priority
- **Windows: `%PROGRAMDATA%\SaveLocker` ACLs on a multi-user box.** The local API token (`api-token`, 0600 on Linux) has **no POSIX mode on Windows** — it inherits the ACL of `%PROGRAMDATA%\SaveLocker`, which the installer widens to give the interactive user Modify. On a machine with several local users, another user may be able to read it and drive this machine's agent. Note this is **not a new exposure**: `config.json` in the same directory already holds the long-lived machine key under the same ACL. Fix both together — tighten the directory ACL to the enrolling user + SYSTEM, or move mutable per-user state out of the machine-wide directory. `run-local-api-tests.ps1` only asserts the file *exists* on Windows; give it a real ACL assertion once the model is decided.
- **Linux agent secret permissions and state layout.** `config.json` contains a long-lived machine key, but file privacy currently depends on the launching shell's umask. Enforce `0700` on private state directories and `0600` on config, queue, health, and log files in code, including CLI enrollment paths. Consider separating immutable app files from mutable XDG config/state so upgrades and permission repair cannot overlap the executable tree.
- **Harden the `systemd --user` unit.** Keep the non-root per-user design, but add and Deck-test conservative restrictions such as `UMask=0077`, `NoNewPrivileges=yes`, `PrivateTmp=yes`, `ProtectSystem=full`, `RestrictAddressFamilies=AF_UNIX AF_INET AF_INET6`, `RestrictSUIDSGID=yes`, and `LockPersonality=yes`. Mirror every directive in both `packaging/linux/savelocker.service` and `SystemdAutoStart.UnitFile()` so the UI toggle cannot regenerate a weaker unit. Do not add `ProtectHome` (save access), `ProtectProc` (Linux writer probe), or `MemoryDenyWriteExecute` (.NET JIT). Record `systemd-analyze --user security savelocker.service` before/after on a Deck.
- **Linux release provenance.** Pin GitHub Actions in `release.yml` to full commit SHAs, especially the release-upload action with `contents: write`; publish SHA-256 checksums and a GitHub artifact attestation for the tarball; enable immutable releases using a draft → attach all assets → publish flow. Document the verification command beside the Deck install instructions.
- **Constrain external manifest paths.** The Ludusavi manifest is downloaded from mutable `master`, and expanded templates are not currently proven to remain inside the intended Proton prefix. Pin or integrity-verify an approved manifest revision, canonicalize resolved paths, reject `..`/symlink escapes outside allowed roots, and test a hostile manifest entry. Preserve explicit manually mapped portable-save paths as a separate trusted-user path.
- **Registry-based saves** — the Ludusavi manifest has a `registry:` section; currently only `files:` paths are handled.
- **Multi-directory saves** — some games list multiple save paths in the manifest. The sync engine tracks one `SaveDirectory` per game; multi-dir support needs a schema change.
