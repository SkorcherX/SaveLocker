# v0.3.0

_Released 19 July 2026_

Setting up a Steam Deck no longer needs a terminal — plus four bugs that only showed up once the
agent spent an afternoon on real hardware.

## New

- **Folder browser in the agent UI.** Setting a save path on a Steam Deck no longer requires typing
  one into a terminal. The browser is rooted at your home folder and the Steam libraries, and
  includes `/run/media`, so SD cards are reachable.
- **The native dialog still wins where there is one.** On Windows the agent opens Explorer as before;
  the in-app browser only appears where there is no desktop file picker.
- **Inline save-path editing in the console.** Each machine's path is edited in its own row, labelled
  with the machine's name. Where an agent has reported a guess, the row offers one-click **Apply**.
- **`scan` offers to map an unmapped game** instead of just listing it. Prompting is skipped when
  input is redirected, so systemd and scripts never block.

## Fixed

- **`install.sh` destroyed a running agent and then printed "Installed."** This is the documented
  Linux update path, so it affected every Deck upgrade. Stop the agent before upgrading *to* this
  version from v0.2.0; from here on the script handles it.
- **`savelocker status` failed with 401 on any server that has an admin password set.** It worked on
  a fresh server, which is exactly why it went unnoticed.
- **A stale Proton prefix synced silently.** Renaming or re-pointing a non-Steam shortcut makes Steam
  create a fresh prefix; the old one still exists and still reads fine, it just is not where the game
  writes any more. `doctor` now compares the two.
- **A save folder mapped one level too deep nested itself on restore** — and the pull *succeeded*, so
  nothing looked wrong. The agent now flags a repeated path tail.

## Known issues

- **`savelocker daemon --lan` is removed** and now exits non-zero. Reach a remote agent over an SSH
  tunnel instead: `ssh -L 5178:localhost:5178 <user>@<host>`.
- Two machines whose saves have diverged will keep reporting a conflict until it is resolved in the
  console. Keeping both agents on the same version avoids most of these.
