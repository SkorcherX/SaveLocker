# v0.3.4

_Released 23 July 2026_

A correctness release for conflicts. Everything here fixes behaviour that was already supposed to
work — nothing new to learn, and nothing to reconfigure.

**This is a server-side release.** Update the container; your agents do not need updating.

## Fixed

- **One conflict no longer becomes dozens.** While a conflict was unresolved, every further save from
  that machine recorded a *separate* conflict. One weekend of playing produced **75 conflicts for a
  single disagreement**. They now fold into one, which counts how many saves are involved and names
  the machine that is stuck.

- **The console showed you the wrong conflict.** When several existed for a game it displayed exactly
  one — the **oldest** — so the save you had actually been playing sat in a conflict the screen never
  drew. Resolving the one on offer restored day-old progress. Every conflict is now listed, most
  recent first.

- **Old saves were never cleaned up on a game with a conflict.** Version retention only ran after a
  save was accepted, and a conflicted save never is. One game reached **80 stored versions and
  2.6 GB** while set to keep 5. Retention now runs regardless, and there is a **Prune now** button
  for applying it immediately.

- **Resolving a conflict did not reach the machines.** It updated the server and nothing else, so the
  next save from either machine conflicted all over again — including from the machine whose save you
  just chose. Both machines are now told to pull.
  - The pull is deliberately cautious: a machine holding changes it has never uploaded is **not**
    overwritten. It reports that it is blocked instead, and you decide.

- **Nothing warned you before you chose.** Picking a version to keep is the most consequential button
  in the console and it acted immediately, with no confirmation. It now tells you what will happen —
  including how many newer saves will stop being the one machines pull.

- **Conflict alerts could be dismissed as if that fixed something.** Every other alert clears itself
  once the machine recovers; a conflict does not — it waits for a person. Dismissing it looked
  identical to resolving it. Conflict alerts now offer **Resolve** instead.

## New

- **Download a save from the console.** Any version can be downloaded from the game's Versions list.
  Previously the only copy lived on the server and on the machines themselves, so there was no way to
  set one aside before doing something irreversible.
- **Prune now**, on the Versions list, applies the game's retention limit without waiting for the next
  save. The current Latest and anything in an open conflict are never removed.
- Conflicts show **which machine is stuck**, and each option shows its machine, time and size instead
  of an id.

## Upgrading

```
docker compose pull && docker compose up -d
```

The database updates itself on start. ⚠️ Because of that update, **rolling back to 0.3.3 or earlier is
not supported** — take a copy of the database first if you want the option.

Agents are unaffected and need no action. If a game is stuck in a conflict loop right now, upgrade
first, then resolve it once in the console; the machines will be told to pull automatically.

## Known issues

- **Steam Deck: the D-pad does not work** in the agent's folder browser. SteamOS Desktop Mode maps
  the right stick to the cursor and the left stick to scroll, and the D-pad to neither. Use
  point-and-click, or arrow keys with a keyboard attached.
- **Linux and Steam Deck have no auto-update.** Download the newer tarball and re-run `install.sh`.
  (Not needed for this release — it changes only the server.)
- **There is no "keep both" option** when resolving. The version you do not choose is kept and can be
  promoted later with **Set as Latest**, but it is not protected from retention.
